# README / ТЗ для агента

## 1. Мета

Потрібно реалізувати Python desktop app для інтерактивного дослідження фазового портрета клиновидного більярда в полі за статтею `main.pdf`.

Програма має бути не просто viewer, а дослідницьким інструментом, у якому основним об'єктом є дискретна орбіта стану:

`(d, tau, i)`

де:
- `d`, `tau` — фазові координати
- `i` — номер активної стінки

Усі візуалізації мають бути синхронними представленнями одних і тих самих орбіт.

## 2. Ключова архітектурна ідея

Програма має бути побудована за принципом:

- одне математичне ядро обчислює орбіти
- UI лише відображає ці орбіти в різних координатах

Три головні synchronized views:
- фазова панель для `wall 1`
- фазова панель для `wall 2`
- геометрична панель більярда в клині

Окремо має бути:
- панель простору параметрів `(alpha, beta)`
- панель параметрів, списку траєкторій і керування

## 3. Технічний стек

Рекомендований стек:
- Python 3.11+
- PySide6 для GUI
- matplotlib або pyqtgraph для графічних панелей
- numpy для обчислень
- pydantic або dataclasses для моделей конфігів і сесій
- yaml/json для конфігів і сесій

Перевага: GUI має бути модульним, з чітким розділенням на model / view / controller.

## 4. Геометричні параметри клина

Допустима область параметрів:
- `0 < alpha <= pi/2`
- `alpha < beta < pi - alpha`

Це робиться для усунення симетрично еквівалентних більярдів відносно осі `Y`.

### Формат вводу кутів

У внутрішніх обчисленнях кути зберігаються у радіанах.

У GUI поля вводу мають підтримувати два формати:
- десятковий запис у радіанах, наприклад `0.5235987756`
- символьний запис через `pi`, наприклад:
    - `pi/6`
    - `pi/3`
    - `2*pi/5`
    - `0.7*pi`

Після парсингу значення зберігається як float у радіанах.

## 5. Основні панелі інтерфейсу

### 5.1 Верхній ряд: дві фазові панелі

Має бути дві окремі панелі фазового простору `(d, tau)`:
- ліва — для старту з `wall 1`
- права — для старту з `wall 2`

Вимоги:
- на панелі показується допустима область визначення
- клік дозволений лише всередині допустимої області
- після лівого кліка створюється нова траєкторія
- стартова точка траєкторії визначається координатами кліку
- активна стінка визначається панеллю, по якій клікнули
- побудова запускається автоматично
- довжина орбіти у фазовому просторі задається параметром `N_phase`
- усі траєкторії мають мати різні стабільні кольори

### 5.2 Нижня ліва панель: геометричний більярд у клині

Має відображатися геометрична траєкторія в самому клині:
- дві стінки клина
- траєкторія у вигляді параболічних сегментів між відбиттями
- точки відбиття
- підсвічування поточного сегмента в режимі replay

Геометрія не є окремим симулятором — вона відновлюється з уже побудованої орбіти `(d, tau, i)`.

Показується лише перші `N_geom` відбиттів.

### 5.3 Нижня права панель: діаграма простору параметрів `(alpha, beta)`

Потрібна окрема 2D-діаграма, де:
- показується допустима область для кутів
- поточна точка `(alpha, beta)` відображається явно
- зміна точки кліком змінює параметри системи
- ручний ввід `alpha`, `beta` теж переміщує точку
- на діаграмі можна накладати спеціальні області з конфігу

Потрібна двостороння синхронізація:
- GUI fields -> point on diagram
- click on diagram -> GUI fields

### 5.4 Бокова панель параметрів і керування

Панель має містити:

#### Траєкторії
Список або таблицю всіх траєкторій:
- `id`
- `wall`
- `d0`
- `tau0`
- `color`
- `visible`
- `selected`

Поведінка:
- траєкторій може бути багато
- одну траєкторію можна зробити вибраною
- вибрана траєкторія використовується для replay за замовчуванням
- по умовчанню вибраною є перша створена траєкторія
- можна сховати або показати окремі траєкторії

#### Параметри
Поля для редагування:
- `alpha`
- `beta`
- `N_phase` — довжина орбіти, по умовчанню 100
- `N_geom` — кількість відбиттів, що показуються в клині

#### Керування
Кнопки:
- `Replay selected`
- `Replay all`
- `Pause`
- `Resume`
- `Step`
- `Reset replay`
- `Export PNG`
- `Save session`
- `Load session`
- `Clear selected trajectory`
- `Clear all trajectories`

## 6. Логіка побудови траєкторій

### 6.1 Додавання траєкторії

Після кліку на одній із фазових панелей:
- перевіряється належність точки допустимій області
- створюється новий seed
- seed має поля:
    - `id`
    - `wall_start`
    - `d0`
    - `tau0`
- запускається обчислення орбіти довжини `N_phase`
- обчислюються також геометричні сегменти для перших `N_geom` кроків

### 6.2 Перебудова

Зміна `alpha` або `beta`:
- автоматично перебудовує всі наявні траєкторії
- автоматично оновлює всі панелі

Зміна `N_phase` або `N_geom`:
- теж перебудовує всі потрібні представлення

## 7. Replay

Режими replay:
- `selected only`
- `all trajectories`

Поведінка:
- у фазових панелях і в клині анімація має йти синхронно
- швидкість replay задається в конфігу
- у режимі `selected only` інші траєкторії можуть лишатися тьмяним фоном
- replay має працювати покроково по орбіті

## 8. Експорт PNG

Потрібен інструмент експорту PNG.

Режими експорту:
- увесь layout
- тільки phase panel 1
- тільки phase panel 2
- тільки wedge view
- тільки `(alpha, beta)` diagram

Стилі експорту:
- color
- monochrome

### Monochrome режим

Чорно-білий режим не має бути простим grayscale від готової картинки.

Потрібно закласти окремий режим рендерингу:
- чорний / сірий / штрихування
- різні типи ліній для різних траєкторій
- контрастні маркери

Це важливо, щоб багато траєкторій були розрізнювані у чорно-білому вигляді.

## 9. Збереження та завантаження сесії

Потрібно підтримати save/load session.

Сесія має містити:
- `alpha`
- `beta`
- усі seeds траєкторій
- visible state для траєкторій
- selected trajectory id
- `N_phase`
- `N_geom`
- поточні налаштування replay
- опційно viewport фазових панелей

Рекомендований формат: JSON або YAML.

## 10. Механізм областей на діаграмі `(alpha, beta)`

Потрібно одразу закласти механізм для відображення спеціальних областей на діаграмі параметрів.

Не зберігати області лише як намальовані полігони.

Базовий формат області:
- `name`
- `label`
- `predicate`
- `style`
- `priority`

### Приклади predicate
- `beta > 2*alpha`
- `beta < pi/2`
- `f(alpha, beta) < 0`
- логічні комбінації через `and` / `or`

### Приклади style
- fill color
- hatch
- opacity
- border color
- line style

Має бути можливість у майбутньому описувати області через явні рівняння або через викличні функції.

## 11. Boundary policy

Цей розділ обов'язковий.

Треба явно зафіксувати правила роботи на межах:
- epsilon для перевірки належності допустимій області
- поведінка при кліку майже на межі
- поведінка для майже вироджених траєкторій
- поведінка біля меж розгалужень відображення
- політика для випадків, коли траєкторія стає нефізичною або чисельно нестійкою

Рекомендація:
- вводити глобальний `eps`
- логувати причину invalid state
- не падати з exception, а позначати орбіту як invalid

## 12. Інспектор вибраної траєкторії

Бажаний окремий інспектор, де показується:
- номер кроку
- активна стінка
- поточні `d`, `tau`
- наступні `d_next`, `tau_next`
- тип переходу, якщо він виділяється окремо
- статус валідності

Це корисно для дослідницького аналізу і дебагу.

## 13. Формат конфігу

Рекомендовано YAML.

Чернетка:

```yaml
app:
  title: Wedge Field Billiard Explorer
  theme: light

simulation:
  n_phase_default: 100
  n_geom_default: 8
  eps: 1.0e-9

replay:
  delay_ms: 120
  selected_only_by_default: true

export:
  dpi: 200
  default_mode: color
  monochrome_line_styles:
    - solid
    - dashed
    - dotted
    - dashdot

view:
  show_grid: true
  show_labels: true
  show_directrix: false
  show_reflection_points: true

regions:
  - name: example_region
    label: Example
    predicate: beta > 2*alpha
    style:
      fill: "#cccccc"
      alpha: 0.3
      hatch: "/"
      border: "#333333"
    priority: 10
```

## 18. Додаткові зауваження для агента

- Не змішувати математичне ядро з кодом UI.
- Усі чисельні константи винести в конфіг.
- Не хардкодити області `(alpha, beta)` — вони мають приходити з конфігу.
- Усі траєкторії мають мати стабільні ідентифікатори.
- При зміні параметрів клина всі траєкторії мають перебудовуватися автоматично.
- PNG export і session save/load треба одразу закласти в архітектуру, навіть якщо спочатку вони будуть мінімальними.
- Код має бути готовий до подальшого розширення, а не одноразовим прототипом.
