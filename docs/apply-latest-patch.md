# apply-latest-patch.sh

`tools/apply-latest-patch.sh` — локальна утиліта для застосування патчів у git-репозиторії.

Основний сценарій використання:

```bash
 tools/apply-latest-patch.sh -d /mnt/c/Users/Lenovo/Downloads/
```

Скрипт шукає останній `.patch` або `.diff` файл у вказаній директорії, пробує застосувати його до поточної git-репи й, якщо пряме застосування не проходить, використовує fallback-стратегії для конфліктних випадків.

## Для чого потрібна утиліта

Утиліта закриває типовий цикл роботи з патчами:

1. ChatGPT або інший інструмент генерує patch-файл.
2. Patch-файл потрапляє у директорію завантажень.
3. Скрипт знаходить останній patch.
4. Скрипт застосовує patch до поточної репи.
5. Якщо patch не застосовується напряму, скрипт пробує fallback-режими.
6. Якщо виникають конфлікти, вони залишаються в робочому дереві для ручного розв’язання.

Це зменшує кількість ручних дій і прибирає ситуації, коли випадково застосовується не той файл або patch доводиться вручну копіювати з чату.

## Базове використання

Застосувати останній patch із директорії:

```bash
 tools/apply-latest-patch.sh -d /mnt/c/Users/Lenovo/Downloads/
```

Запустити з докладним виводом:

```bash
 tools/apply-latest-patch.sh -v -d /mnt/c/Users/Lenovo/Downloads/
```

Тільки показати, який patch буде обрано, але нічого не застосовувати:

```bash
 tools/apply-latest-patch.sh -n -d /mnt/c/Users/Lenovo/Downloads/
```

Застосувати конкретний patch, якщо підтримується опція `-p`:

```bash
 tools/apply-latest-patch.sh -p /mnt/c/Users/Lenovo/Downloads/fix.patch
```

## Як обирається patch

Якщо передано `-d DIR`, скрипт шукає у цій директорії файли:

- `*.patch`
- `*.diff`

Після цього вибирається найновіший файл.

Якщо директорію не передано, типовий порядок такий:

1. `patches/`, якщо така директорія існує;
2. поточна директорія.

## Основна стратегія застосування

Бажана логіка роботи така:

1. Обрати patch.
2. Спробувати чисте застосування через `git apply --check`.
3. Якщо перевірка проходить — застосувати patch.
4. Якщо чисте застосування не проходить — спробувати `git apply --3way`.
5. Якщо 3-way застосування створює конфлікти — залишити їх у робочому дереві.
6. Якщо 3-way не стартує, можна використати merge/worktree fallback, якщо він реалізований у поточній версії скрипта.

Головний принцип: якщо patch не може бути застосований чисто, але git може побудувати конфліктний стан, скрипт не має мовчки відкотити все. Конфлікти краще залишити користувачу для ручного розв’язання.

## Конфлікти

Якщо після fallback-застосування з’явилися конфлікти, треба діяти як після звичайного git merge:

```bash
 git status
```

Далі відкрити конфліктні файли, розв’язати conflict markers, після чого:

```bash
 git add <resolved-files>
 git commit
```

Якщо потрібно скасувати невдале застосування, залежно від ситуації можна використати:

```bash
 git merge --abort
```

або вручну повернути стан через:

```bash
 git reset --hard
```

Перед `git reset --hard` треба переконатися, що в робочому дереві немає важливих незбережених змін.

## Auto-stash

Скрипт може автоматично прибирати локальні зміни у stash перед застосуванням patch.

Типова поведінка:

- tracked-зміни можуть бути застешені автоматично;
- untracked-файли додаються до stash тільки за явною опцією, якщо вона підтримується.

Корисні опції:

```bash
-s   примусово використати stash
-S   не використовувати auto-stash
-U   включити untracked-файли у stash
```

Після успішного застосування patch скрипт може нагадати, що попередні зміни залишились у stash. Їх можна повернути командою:

```bash
 git stash pop
```

## Auto-commit

Якщо увімкнено auto-commit, скрипт після успішного застосування patch може автоматично створити commit.

Типові опції:

```bash
-C          зробити commit після застосування
-c "msg"    використати власний commit message
```

Якщо patch застосувався з конфліктами, auto-commit не має виконуватись до ручного розв’язання конфліктів.

## Strict mode

Strict mode означає: дозволити тільки чисте застосування patch.

У цьому режимі fallback-стратегії не використовуються. Якщо `git apply --check` не проходить, скрипт завершується з помилкою.

Приклад, якщо опція підтримується:

```bash
 tools/apply-latest-patch.sh --strict -d /mnt/c/Users/Lenovo/Downloads/
```

## Рекомендований workflow

Для звичайної роботи:

```bash
 tools/apply-latest-patch.sh -d /mnt/c/Users/Lenovo/Downloads/
```

Для перевірки, який patch буде застосовано:

```bash
 tools/apply-latest-patch.sh -n -d /mnt/c/Users/Lenovo/Downloads/
```

Для складного patch, де можливі конфлікти:

```bash
 tools/apply-latest-patch.sh -v -d /mnt/c/Users/Lenovo/Downloads/
```

Для безпечного режиму без fallback:

```bash
 tools/apply-latest-patch.sh --strict -d /mnt/c/Users/Lenovo/Downloads/
```

## Зв’язок із Пакпатчем

`apply-latest-patch.sh` добре лягає на workflow Пакпатча:

1. Створюється `chatgpt-pack-*` архів.
2. ChatGPT працює всередині disposable repo.
3. ChatGPT повертає `.patch` файл.
4. Користувач завантажує patch.
5. `apply-latest-patch.sh` застосовує останній patch із директорії завантажень.

Таким чином ChatGPT генерує тільки `git diff`, а локальний скрипт детерміновано застосовує цей diff до реальної репи.

## Що важливо пам’ятати

- Скрипт треба запускати всередині git-репозиторію, до якого застосовується patch.
- Перед застосуванням складних patch-файлів бажано подивитись `git status`.
- Якщо є важливі локальні зміни, краще зробити commit або stash вручну.
- Якщо patch застосувався з конфліктами, це нормальний стан: їх треба розв’язати як звичайні merge-конфлікти.
- Якщо patch був згенерований через Пакпатч, він має відповідати реальним шляхам у репі.
